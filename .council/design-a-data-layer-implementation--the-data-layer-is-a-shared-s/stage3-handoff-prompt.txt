## Revised Plan
## 1. Goal and constraints
- **Goal**: Design a unified, shared data layer embedded within the `layers` codebase to serve as the single source of truth for all Caelator ecosystem plugins (`council`, `openclaw-pm`, `research-radar`, `evolve`).
- **Constraint**: Must strictly use LanceDB as the underlying vector and structured data store for shared, queryable state. No alternative databases may be considered for this purpose.
- **Constraint**: The data layer must support both semantic (vector) search and structured (relational/metadata) queries natively.
- **Constraint**: Plugins must cease maintaining isolated persistent storage silos for shared state; the design must prevent data duplication across the ecosystem.
- **Constraint**: This phase is strictly for architectural design and documentation; no Rust implementation or plugin refactoring is to be executed yet.

## 2. Candidate approaches
### Approach A: Single Monolithic Collection (The "Everything Bagel")
- **Concept**: A single LanceDB collection (`caelator_memory`) stores every entity across all plugins with generic metadata plus a flexible payload blob.
- **Tradeoffs**:
  - *Pros*: Trivial cross-plugin semantic search. Minimal collection management overhead.
  - *Cons*: Sparse schema, weak typing, heavy serialization logic.

### Approach B: Domain-Specific Collections with a Unified Relational Graph
- **Concept**: Fixed, strongly-typed domain collections share a mandatory `MemoryHeader` (`urn`, `parent_urn`, `workspace_id`, `tags`, `origin_plugin`), with embeddings stored inline.
- **Tradeoffs**:
  - *Pros*: Strong Rust typing. Cleaner per-domain schema evolution.
  - *Cons*: Multi-collection semantic aggregation required. Inline vectors complicate re-embedding.

### Approach C: Hybrid (LanceDB for Shared State/Vectors + File-based Append Logs)
- **Concept**: Use strongly-typed LanceDB collections (`entities`, `relations`, `artifacts`, `embeddings`, `projections`) plus filesystem append logs for pure audit streams.
- **Tradeoffs**:
  - *Pros*: Keeps shared queryable state centralized while avoiding misuse of LanceDB for temporal traces; dedicated embeddings table preserves domain typing and simplifies re-indexing.
  - *Cons*: Requires an explicit classification rule and cross-boundary lifecycle policy.

## 3. Recommended approach with rationale [CHANGED]
**Recommended: Approach C (Hybrid with Dedicated Embedding Index and File-based Logs)**
- **Rationale**: Keep all shared, queryable, cross-plugin state in LanceDB; keep data filesystem-local **only if** it is append-only, never queried cross-plugin, and not semantically searchable.
- **Classification Tiebreaker**: If any predicate is uncertain, default to `LanceDB canonical`; the ADD must include at least one borderline worked example.
- **Dedicated Embeddings Physical Design**: Keep a logical `SemanticIndex`, but physically store vectors in one collection per registered embedding space/dimension (for example `embeddings_rr_text_v1`), with `space_id` as a registry concept rather than a mixed-dimension single table.
- **Greenfield Vector Spaces**: `research-radar` is migrating from SQLite keyword search, not from an existing embedding store; embedding space/model selection is therefore a V1 prerequisite, not a migration afterthought.
- **Unified Memory Language**: Every canonical row carries a standard `MemoryHeader`, and file logs may reference URNs but never replace canonical state.

## 4. V1 scope [CHANGED]
- **Deliverable**: A comprehensive markdown Architecture Design Document (ADD).
- **Schema Definition**: Define canonical tables and ownership rules:
  - `entities`: canonical stateful nodes keyed by `urn`, plus indexed `domain` and `kind` columns derived at write time.
  - `relations`: typed edges between URNs.
  - `artifacts`: larger content blobs or external-file descriptors attached to URNs.
  - `embeddings_*`: semantic index collections keyed by `(subject_urn, chunk_id)` within each registered `space_id`; the ADD must define the space registry, dimensionality, and model provenance fields.
  - `projections`: derived, rebuildable read models/materializations keyed by `(projection_name, subject_urn, version)` with `built_from_version`; stale projections must be rejected or lazily rebuilt.
- **Identity Spec**: The ADD must define URNs as `urn:caelator:{plugin_or_domain}:{kind}:{id}` with global uniqueness, allowed character rules, and the rule that URNs are opaque for identity logic but parseable for diagnostics.
- **Schema Evolution**: Promote this into V1: additive columns are the default, each collection has a monotonic schema version, `layers_data::migrate()` upgrades stores in place, and plugins declare a supported schema-version range rather than assuming HEAD.
- **Tombstone Lifecycle**: Define soft-delete retention, the precondition for reaping (no protected audit-log reference within the retention window), and a bounded compaction policy so tombstones do not grow without limit.
- **API Surface Design**: Define a typed Rust crate (`layers-data`) exposing `EntityStore`, `RelationStore`, `ArtifactStore`, `SemanticIndex`, `EventLog`, and `QueryPlanner`.
  - `QueryPlanner` is only for cross-collection resolution and hybrid structured+semantic queries; simple CRUD/filtering stays in the store traits.
  - Entity/artifact writes must synchronize embedding upserts/tombstones through the same API boundary.
  - Write concurrency is mediated by one crash-safe mechanism inside `layers-data` (lease/stale-lock recovery or equivalent single-writer contract), not an unspecified mutex.
  - Bulk re-embedding must use chunked lock-yielding writes or shadow-build plus atomic cutover; it may not hold the writer slot for the entire rebuild.
- **Migration Strategy**: Include a concrete ETL matrix for every persisted artifact in every plugin, and classify each as `LanceDB canonical` or `filesystem audit` using the rule above.

## 8. Control flow [CHANGED]
1. **Initialization**: Plugins call `layers_data::open(global_store_path, plugin_context)` and receive typed capability handles.
2. **Schema + Compatibility Check**: `layers-data` verifies collection schema versions, runs `layers_data::migrate()` if needed, and rejects unsupported plugin/schema version combinations.
3. **Identity & Namespace**: Every write includes `PluginContext { plugin_id, workspace_id, capabilities }` and a generated canonical URN plus stored `domain`/`kind` fields.
4. **Write Path**: A plugin writes through `EntityStore`/`ArtifactStore`; `layers-data` acquires the crash-safe writer lease, persists canonical rows, and synchronizes embedding rows or tombstones in the same mediated operation.
5. **Read Path**: `QueryPlanner` composes structured filters plus `SemanticIndex` hits across registered embedding-space collections, resolves `subject_urn` back to canonical records, and refuses or rebuilds stale projections before returning normalized `MemoryNode` results.
6. **Audit Path**: Pure append-only traces are written to workspace-local logs and may reference URNs, but canonical records they reference are tombstoned until tombstone-retention and reference-check rules permit reaping.
