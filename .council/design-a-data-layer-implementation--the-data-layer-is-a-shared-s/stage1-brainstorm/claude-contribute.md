1. **Section: 2. Candidate approaches**
   - **Suggest:** Add a third approach — "Hybrid: LanceDB for vector-bearing collections, retain file-based append logs for event streams." Council's JSONL traces and openclaw-pm's event logs are append-only audit trails that gain nothing from vector indexing. Forcing them into LanceDB collections adds schema complexity for data that is never queried semantically.
   - **Why:** Council writes `RunTrace`, `PhaseMetrics`, and `LearningRecord` as JSONL append logs. Openclaw-pm writes `events.jsonl` and `archive.jsonl`. These are temporal audit streams, not semantic retrieval targets. The plan's all-or-nothing stance ("plugins must cease maintaining isolated persistent storage silos") conflates *shared queryable state* with *append-only operational logs*. Exempting pure event streams from the LanceDB mandate simplifies the schema by ~40% while still unifying the entities that actually benefit from cross-plugin search.

2. **Section: 3. Recommended approach with rationale**
   - **Suggest:** Replace the assertion that research-radar "embeds source code" with the actual state: research-radar uses SQLite with keyword `LIKE` matching and has zero embedding infrastructure. The `search_entries` method has a comment reading "In a future phase this will delegate to a proper vector store." The migration is from SQLite, not from an existing LanceDB setup.
   - **Why:** The plan's rationale for separate vector spaces ("research-radar embeds source code, while openclaw-pm embeds natural language tasks") is built on a false premise. Neither plugin produces embeddings today. This changes the migration story significantly — you're not preserving existing vector spaces, you're introducing them for the first time, which means you can choose a single embedding model from the start and sidestep the cross-model compatibility problem entirely.

3. **Section: 4. V1 scope**
   - **Suggest:** Replace the abstract schema names (`Nodes`, `Edges`, `Documents`, `Tasks`) with names derived from the actual domain types that exist today. Concretely: `sources` + `entries` + `profiles` (from research-radar's 9 SQLite tables), `project_state` + `tasks` (from openclaw-pm's YAML state), `run_traces` + `plans` + `learnings` (from council's JSONL records). Map the real structs first, then abstract.
   - **Why:** The current schema names are invented top-down without reference to any existing data structure. Research-radar has `Source`, `Entry`, `Profile`, `ScanJob`, `Subscription`, `ScoredMatch` — none of which map cleanly to "Nodes" or "Documents." Starting from concrete types and finding the natural shared abstractions (rather than imposing generic graph vocabulary) will produce a schema plugins can actually adopt without awkward impedance-mismatch wrappers.

4. **Section: 8. Control flow**
   - **Suggest:** Replace the `DataLayerHandle` broker pattern with a simpler model: `layers` exposes a typed Rust crate (`layers-data`) that plugins depend on directly as a library. Plugins call `layers_data::open(path)` and get back typed table handles. No "binding," no "requesting a handle from the host," no runtime brokering.
   - **Why:** Every existing plugin already uses direct library calls for storage — research-radar calls `DbPool::init()`, openclaw-pm calls `fs2::File::open()`, council calls `std::fs::OpenOptions`. The plan introduces a runtime plugin-host negotiation pattern (`plugin requests DataLayerHandle from layers host`) that doesn't match how any of these plugins actually load. A compile-time crate dependency is simpler, type-safe, and idiomatic Rust. The "host grants handle" pattern only makes sense for dynamically loaded WASM plugins, which are out of scope.

5. **Section: 9. Risks and open questions**
   - **Suggest:** Add a risk: "Openclaw-pm uses file locking (`fs2` crate) with exclusive locks and atomic temp-file-rename writes for state mutations. LanceDB's concurrency model must be validated against this pattern — if LanceDB doesn't support single-writer file locking semantics, openclaw-pm's atomicity guarantees will silently degrade."
   - **Why:** Openclaw-pm's `state.lock` mechanism and atomic `write_temp → fsync → rename` pattern are load-bearing correctness features, not incidental. The plan doesn't acknowledge that any existing plugin has concurrency controls, and LanceDB's MVCC behavior is different from exclusive file locks. This is a concrete compatibility hazard that should be called out.

6. **Section: 7. Files, binaries, and storage**
   - **Suggest:** Change `~/.caelator/store/*.lance` to a path that respects the existing convention. Research-radar stores data at `~/.research-radar/data.db`, openclaw-pm at `.openclaw/pm/` (workspace-relative). The design should specify whether storage is user-global (`~/.caelator/`) or workspace-relative, and document the migration for both `~/.research-radar/data.db` and `.openclaw/pm/state.yaml`. Currently the plan only mentions one path and ignores that openclaw-pm's storage is project-scoped, not user-scoped.
   - **Why:** This is a hidden semantic conflict. Research-radar's data is global (all research across all projects in one DB). Openclaw-pm's data is per-workspace (each project has its own `.openclaw/pm/`). Forcing both into a single `~/.caelator/store/` either loses openclaw-pm's project isolation or requires a workspace-keying scheme that the plan doesn't address. This needs an explicit decision.

7. **Section: 4. V1 scope (Migration Strategy)**
   - **Suggest:** The migration strategy should be a mechanical SQLite-to-LanceDB table-by-table ETL for research-radar's 9 existing tables, not a conceptual "step-by-step documented path." Specify exactly: `sources` → `rr_sources` LanceDB table (5 columns, no vector), `entries` → `rr_entries` table (add embedding column, backfill with chosen model), `profiles` → `rr_profiles` (deserialize JSON keyword arrays into Arrow list columns). The 6 remaining tables (`queries`, `results`, `scan_jobs`, `subscriptions`, `item_scores`, and indexes) need explicit dispositions — migrate, drop, or defer.
   - **Why:** Research-radar has 1020 lines of storage code with 25+ methods across 9 tables. A vague "migration path" will stall at implementation time when someone discovers that `item_scores` has a composite unique constraint (`entry_id, profile_id`) that needs a LanceDB equivalent, or that `scan_jobs` tracks ephemeral state that probably shouldn't be in a shared data layer at all. Concrete table-level decisions now prevent blocking questions later.
