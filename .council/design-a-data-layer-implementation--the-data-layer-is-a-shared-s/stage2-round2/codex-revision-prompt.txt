Severity-proportional disposition (IMPORTANT — follow strictly):

## Revised Plan
## 1. Goal and constraints
- **Goal**: Design a unified, shared data layer embedded within the `layers` codebase to serve as the single source of truth for all Caelator ecosystem plugins (`council`, `openclaw-pm`, `research-radar`, `evolve`).
- **Constraint**: Must strictly use LanceDB as the underlying vector and structured data store for shared, queryable state. No alternative databases may be considered for this purpose.
- **Constraint**: The data layer must support both semantic (vector) search and structured (relational/metadata) queries natively.
- **Constraint**: Plugins must cease maintaining isolated persistent storage silos for shared state; the design must prevent data duplication across the ecosystem.
- **Constraint**: This phase is strictly for architectural design and documentation; no Rust implementation or plugin refactoring is to be executed yet.

## 2. Candidate approaches
### Approach A: Single Monolithic Collection (The "Everything Bagel")
- **Concept**: A single LanceDB collection (`caelator_memory`) stores every entity across all plugins with generic metadata plus a flexible payload blob.
- **Tradeoffs**:
  - *Pros*: Trivial cross-plugin semantic search. Minimal collection management overhead.
  - *Cons*: Sparse schema, weak typing, heavy serialization logic.

### Approach B: Domain-Specific Collections with a Unified Relational Graph
- **Concept**: Fixed, strongly-typed domain collections share a mandatory `MemoryHeader` (`urn`, `parent_urn`, `workspace_id`, `tags`, `origin_plugin`), with embeddings stored inline.
- **Tradeoffs**:
  - *Pros*: Strong Rust typing. Cleaner per-domain schema evolution.
  - *Cons*: Multi-collection semantic aggregation required. Inline vectors complicate re-embedding.

### Approach C: Hybrid (LanceDB for Shared State/Vectors + File-based Append Logs)
- **Concept**: Use strongly-typed LanceDB collections (`entities`, `relations`, `artifacts`, `embeddings`, `projections`) plus filesystem append logs for pure audit streams.
- **Tradeoffs**:
  - *Pros*: Keeps shared queryable state centralized while avoiding misuse of LanceDB for temporal traces; dedicated embeddings table preserves domain typing and simplifies re-indexing.
  - *Cons*: Requires an explicit classification rule and cross-boundary lifecycle policy.

## 3. Recommended approach with rationale [CHANGED]
**Recommended: Approach C (Hybrid with Dedicated Embedding Index and File-based Logs)**
- **Rationale**: Keep all shared, queryable, cross-plugin state in LanceDB; keep data filesystem-local **only if** it is append-only, never queried cross-plugin, and not semantically searchable.
- **Dedicated Embeddings Table**: Semantic retrieval flows through `embeddings(subject_urn, space_id, chunk_id, vector, content_hash, status)` so model migration and multi-domain search do not churn domain schemas.
- **Greenfield Vector Spaces**: `research-radar` is migrating from SQLite keyword search, not from an existing embedding store; embedding model/space selection is therefore a V1 prerequisite, not a migration afterthought.
- **Unified Memory Language**: Every canonical row carries a standard `MemoryHeader`, and file logs may reference URNs but never replace canonical state.

## 4. V1 scope [CHANGED]
- **Deliverable**: A comprehensive markdown Architecture Design Document (ADD).
- **Schema Definition**: Define canonical tables and ownership rules:
  - `entities`: canonical stateful nodes keyed by `urn`.
  - `relations`: typed edges between URNs.
  - `artifacts`: larger content blobs or external-file descriptors attached to URNs.
  - `embeddings`: semantic index rows keyed by `(subject_urn, space_id, chunk_id)`.
  - `projections`: derived, rebuildable read models/materializations keyed by `(projection_name, subject_urn, version)`; never the system of record.
- **Identity Spec**: The ADD must define URNs as `urn:caelator:{plugin_or_domain}:{kind}:{id}` with global uniqueness, allowed character rules, and the rule that URNs are opaque for logic but parseable for diagnostics.
- **Schema Evolution**: Promote this into V1: additive columns are the default, each collection has a monotonic schema version, `layers_data::migrate()` upgrades stores in place, and plugins declare a supported schema-version range rather than assuming HEAD.
- **API Surface Design**: Define a typed Rust crate (`layers-data`) exposing `EntityStore`, `RelationStore`, `ArtifactStore`, `SemanticIndex`, `EventLog`, and `QueryPlanner`.
  - `QueryPlanner` is only for cross-collection resolution and hybrid structured+semantic queries; simple CRUD/filtering stays in the store traits.
  - Entity/artifact writes must synchronize embedding upserts/tombstones through the same API boundary.
  - Write concurrency is mediated by one mandated mechanism inside `layers-data` (initial design target: a cross-process write coordinator/lock wrapper so only one writer commits at a time).
- **Migration Strategy**: Include a concrete ETL matrix for every persisted artifact in every plugin, and classify each as `LanceDB canonical` or `filesystem audit` using the rule above.

## 5. V2 / later
- **Implementation**: Writing the actual Rust code in the `layers` codebase to instantiate the LanceDB connection and expose the APIs.
- **Plugin Refactoring**: Updating `research-radar`, `council`, and `openclaw-pm` source code to consume the new `layers` API and deleting their isolated local storage mechanisms.
- **Advanced Graph Queries**: Implementing multi-hop relational queries across LanceDB collections within the `layers` API layer.
- **Data Sync**: Mechanisms for synchronizing local LanceDB storage across different developer machines or to a cloud backup.

## 6. Out of scope / do not build
- Evaluation, benchmarking, or consideration of any database other than LanceDB (e.g., SQLite, Qdrant, Chroma).
- Writing any Rust implementation code for the data layer or modifying existing plugin codebases during this design phase.
- Design of network-level synchronization protocols or multi-user concurrent access models (assume single-user local dev environment for now).
- Designing generic data ingestors for non-Caelator external tools.

## 7. Files, binaries, and storage [CHANGED]
- `docs/architecture/001-unified-data-layer.md`: The canonical output of this phase, detailing schema, API, migration, versioning, and audit-log classification.
- **Canonical Store Decision**: Use one user-global LanceDB store at `~/.caelator/store/*.lance` with mandatory `workspace_id` on every canonical row; workspace-relative storage is reserved for append-only local audit logs only.
- `crates/layers-data/src/`: The directory within the `layers` codebase where schema definitions, capability traits, and LanceDB implementations will live in V2.

## 8. Control flow [CHANGED]
1. **Initialization**: Plugins call `layers_data::open(global_store_path, plugin_context)` and receive typed capability handles.
2. **Schema + Compatibility Check**: `layers-data` verifies collection schema versions, runs `layers_data::migrate()` if needed, and rejects unsupported plugin/schema version combinations.
3. **Identity & Namespace**: Every write includes `PluginContext { plugin_id, workspace_id, capabilities }` and a generated canonical URN.
4. **Write Path**: A plugin writes through `EntityStore`/`ArtifactStore`; `layers-data` acquires the write coordinator, persists canonical rows, and synchronizes embedding rows or tombstones in the same mediated operation.
5. **Read Path**: `QueryPlanner` composes structured filters plus `SemanticIndex` hits, resolves `subject_urn` back to canonical records, and returns normalized `MemoryNode` results.
6. **Audit Path**: Pure append-only traces are written to workspace-local logs and may reference URNs, but canonical records they reference are tombstoned rather than hard-deleted.

## 9. Risks and open questions [CHANGED]
- **LanceDB Rust API Maturity**: Validate that required filtering, joins, and projection rebuild workflows are supportable without excessive raw-engine escape hatches.
- **Write Coordination Feasibility**: Confirm the chosen single-writer mediation pattern is sufficient for expected plugin concurrency and does not make common workflows unusably serial.
- **Schema Evolution Operations**: The versioning policy is defined in V1, but migration ergonomics and rollback strategy still need validation.
- **Embedding Space Governance**: Approved `space_id` values, model provenance fields, and re-embedding triggers must be finalized in the ADD.
- **Projection Discipline**: Ensure projections remain rebuildable caches/read models and never accumulate canonical business state.

## 10. Validation plan [CHANGED]
- **Schema Desk-Check**: Map existing plugin data structures to the canonical schema, including explicit `projections` usage or confirmation that no projection is needed.
- **Migration Mapping Matrix**: For every persisted artifact in `research-radar`, `openclaw-pm`, `council`, and `evolve`, record source store, target canonical rows or audit-log classification, deprecated path, and one worked end-to-end example per plugin.
- **Compatibility Walkthrough**: Include pseudocode for schema-version negotiation, `layers_data::migrate()`, and one non-breaking additive schema change.
- **Integrity Walkthrough**: Include pseudocode for coordinated entity+embedding upsert/delete, tombstoning policy for URNs referenced by file logs, and orphan-vector prevention checks.
- **Concurrency Walkthrough**: Describe the write-mediation contract, conflict model, and a representative multi-plugin write scenario to validate that the design is implementable.
